# 2026-06-02 — Built and shipped lever-runner v0.1.0

## What was built
- **Repo**: `github.com/SuperInstance/lever-runner` (public). Tag `v0.1.0` released.
- **Architecture**: post-inference command execution. LLM produces a 3-8 word intent phrase; never sees shell. Phrase is embedded (sentence-transformers/all-MiniLM-L6-v2), matched against a LanceDB table of pre-approved commands, top-1 above the trust floor is executed in `/tmp/lever-runner/<sid>/` with a 30s timeout.
- **Code**: ~1,400 lines of Python across `src/lever_runner/` (orchestrator, store, intent_extractor, executor, bot, cli, http_api, benchmark, auto_promote, token_logger, seed_export, seed_import) + `tests/smoke.py` (18/18 passing).
- **Seed pack**: 66 commands spanning system health, git, docker, npm, networking, files, services, Ollama, **OpenClaw self-ops** (version, status, logs, config), and **lever-runner self-ops** (version, git log, benchmark, command count, reseed, sandbox).
- **Status**: pushed, tagged, released, bot running under systemd (PID was 97451 at last check), cron installed for hourly self-improvement.

## Design decisions (the "essence" per Casey)
- The LLM is the *phrase compressor*, not the *command inventor*. The phrase is matched against a fixed table. The LLM has no way to inject shell. That's the security property.
- 4 LLM backends: passthrough (no LLM call; user request is the phrase verbatim), minimax (hosted MiniMax-M3 via Anthropic-compatible API), openai, ollama. Default is passthrough.
- Trust is a **gate** (above a floor, you can run), not a tiebreaker for similar matches. Bug fix: original `min(eligible, key=lambda m: (-m.trust_score, m.score))` was inverted — `min` with `-trust` chose the *lowest* trust, not the highest. Replaced with `min(eligible, key=lambda m: m.score)`. Regression test in `tests/smoke.py` step 7.
- No-match semantics: `MATCH_SIMILARITY_FLOOR=0.55` (env-overridable). Below this, `no_match=True` is set and `/teach` is suggested. Was previously triggered only by an empty table.
- Auto-promote rewrites are **gated** on `REMOTE_LLM_API_KEY` being set; no key = no-op (no surprise network calls).
- The "Community Skills" export/import (`seed_export.py`, `seed_import.py`) uses JSONL without embeddings — the importer re-encodes on the receiving machine.

## Bugs caught and fixed during the build
1. `python-telegram-bot==20.8` is broken on Python 3.14 (name-mangling AttributeError in `Updater._Updater__polling_cleanup_cb`). Pinned to `>=21,<23` in `requirements.txt`.
2. `read_consistency_interval=0` (as `timedelta(seconds=0)`) needed in `lancedb.connect()` so /teach → /do finds the new row immediately. LanceDB OSS has eventual-consistency reads by default.
3. `auto_promote.py` was calling `table.to_pandas()` which requires `pandas` (lancedb's to_pandas triggers pyarrow's pandas shim). Switched to `search().limit(N).to_list()`.
4. Trust-selection inversion (above).
5. The `bot.py` did not actually check `ALLOWED_USER_ID` even though the variable was in `.env.example`. Added the check across all 5 handlers (`/do`, `/teach`, `/status`, `/start`, `/help`, plain-text fallback).
6. The systemd unit's ExecStart needed `PYTHONPATH=/home/ubuntu/lever-runner/src` to find the package (no proper `setup.py`/`pyproject.toml` install yet — using a flat `src/` layout).

## v0.2.0 — per-chat trust, LLM timeout, log rotation, /healthz

Built on top of v0.1.1, after Casey said "go" twice. Pushed (commit `e9f527c`), tagged `v0.2.0`, GitHub release live.

### What changed
- **Per-chat trust isolation.** Each chat (Telegram, HTTP, CLI `--chat-id`) gets its own `commands_<chat_id>` LanceDB table. `/teach` in chat A is invisible to chat B. New chats auto-seed with the global pack on first use (batched `embedder.encode()` over all 66 SEED_COMMANDS → ~0.5s vs ~3s one-at-a-time). Legacy `commands` table migrates to `commands_default` on first access. `auto_promote.py` now iterates all chats via `_iter_chat_tables()`.
- **LLM request timeout (5s).** `LLM_TIMEOUT_SEC` env var (default 5). `requests.post(timeout=...)` covers connect+read. `Timeout` and `ConnectionError` → log + fall back to passthrough (raw user request as intent) so a hung provider can't hang the orchestrator. `HTTPError` (4xx/5xx) still propagates — wrong key / service outage should be visible. Tagged `backend="passthrough-fallback"` in the Extraction for transparency.
- **Log rotation for JSONL.** `token_usage.jsonl` and `embed_usage.jsonl` rotate at 5 MiB with up to 3 backups (~20 MiB cap per log). Tunable via `TOKEN_LOG_MAX_BYTES` and `TOKEN_LOG_BACKUP_COUNT`. Hand-rolled `_rotate_if_needed()` rather than `RotatingFileHandler` so we don't hold a long-lived file handle in a stateless helper.
- **`/healthz` endpoint.** Returns `{ok, version, uptime_sec, tables: {name: count}, total_commands, lancedb_path}`. 503 with error JSON if lancedb is unreachable.

## v0.3.0 — DeepInfra backend, provider fallback chain (2026-06-02 16:25 UTC)

Built in response to Casey's frustration with rate limits. v0.3.0
ships the fallback chain so the bot doesn't hang on a 429 from
MiniMax-M3 — it transparently tries DeepInfra, then if that also
fails, falls through to passthrough mode (raw user request as
intent).

### What changed
- **LLM_FALLBACKS env var.** Comma-separated list of backends tried
  in order when the primary errors. Default: `deepinfra`. The
  chain always ends at `passthrough` (last-resort).
- **Retryable status codes:** 408, 425, 429, 500, 502, 503, 504,
  529. Plus Timeout and ConnectionError. Plus HTTP 401 *in chain
  context only* (a bad key for one provider shouldn't break the
  whole chain).
- **DeepInfra backend (`LLM_BACKEND=deepinfra`).** OpenAI-compatible
  endpoint at `api.deepinfra.com/v1/openai`. Default model:
  `meta-llama/Meta-Llama-3.1-8B-Instruct`. Tried the
  `Qwen/Qwen3.5-4B` reasoning model first; it leaves `content`
  empty because the answer goes in `reasoning_content`, so it
  can't be used as a phrase compressor.
- **Per-backend URL/model env vars:** `LLM_MINIMAX_BASE_URL`,
  `LLM_DEEPINFRA_BASE_URL`, `LLM_OPENAI_BASE_URL`, etc. Lets the
  operator override one backend's endpoint without leaking the
  global `LLM_BASE_URL` into fallback attempts.
- **Key resolution bug fix.** Previously, when both `LLM_API_KEY`
  and `DEEPINFRA_API_KEY` were set, the resolver concatenated them
  into a 64-char invalid key (because `or "".join(...)` is not
  short-circuit on empty strings). Now first-non-empty wins.

### Bot config
- `LLM_BACKEND=minimax`, `LLM_FALLBACKS=deepinfra`
- `DEEPINFRA_API_KEY` added to `~/lever-runner/.env` (chmod 600,
  gitignored). The agent's process env also has `MINIMAX_API_KEY`
  and `DEEPINFRA_KEY`; systemd inherits them via
  `EnvironmentFile=`.
- End-to-end test: `do("check disk usage on the server")` →
  minimax 401 → deepinfra `"check server disk usage"` → match
  `"show disk usage"` → `df -h` → ok=True. No more waiting on
  rate limits.

### Tests
- 34/34 → **59/59**. New (20 checks): 8 retryable-status-code
  policy, 4 chain-parsing, 1 end-to-end
  timeout-falls-back-to-passthrough, plus the existing fallback
  tests for key resolution.

## v0.3.1 — more bot commands, full table introspection (2026-06-02 ~16:35 UTC)

Small, focused release. The bot now has three new slash commands
and the library has two new methods. All 90 smoke tests pass.

### /commands [N] [--page=K]
Paginated table listing, sorted by trust desc. Default 20 per
page, max 100. Page 1 of (total+limit-1)//limit, total count
shown. Each row: `trust  succ/fail  phrase  last_run_date`. Lets
the user see "what does this bot know how to do?" without
guessing intent phrases.

### /stats <phrase>
Full stats for one command. Embeds the phrase, picks the best
match, then pulls the full row from `store.get_by_id()` and
shows everything: phrase, command, trust, success/failure counts,
created_at, last_run + last_result, embedding distance and
similarity, and the row id. Respects MATCH_SIMILARITY_FLOOR
(default 0.55) so an unrelated phrase returns "no match" with
the top similarity for context.

### /teach --trust=N
Override starting trust. Validates 0-100 range. Works in bot
and CLI. Parser handles both `--trust=70` and `--trust 70`
forms, and consumes leading flags so the rest of the line still
parses as `"phrase" | command`.

### Library additions
- `CommandStore.list_all(limit, offset, min_trust)` — paginated
  listing. Uses lancedb's `.to_pandas()` for filter + sort.
  Pagination is Python-side because tables are O(hundreds).
- `CommandStore.get_by_id(row_id)` — fetch one row's full
  metadata.
- `orchestrator.list_commands(chat_id, limit, offset, min_trust)`
  — same at the orchestrator level.

### Deps
- `pandas>=2.0` added to `requirements.txt`. Was a transitive
  lancedb dep; newer lancedb doesn't declare it as a hard req
  so fresh installs would have failed the smoke test.
