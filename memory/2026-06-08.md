### Critical Finding: OpenSMILE Already Has What We Need

The pip-installed `opensmile 2.6.0` ships a pre-built `libSMILEapi.so` for ARM64 Linux at:
```
opensmile/core/bin/manylinux_2_17_aarch64/libSMILEapi.so
```

**No custom build or source refactoring is needed.** The entire refactoring plan in `memory/opensmile-refactoring-plan.md` was over-engineered.

### Key Architecture Facts (from codebase analysis)

1. **libSMILEapi.so exposes ~20 C functions** for the full lifecycle:
   - `smile_new()` / `smile_free()` ‚Äî lifecycle
   - `smile_initialize(configFile, options)` ‚Äî loads config .ini
   - `smile_extsource_write_data()` ‚Äî push PCM in
   - `smile_extsink_set_data_callback()` ‚Äî receive features via callback

2. **Python wrapper uses ctypes directly** ‚Äî no Cython, no pybind11, no SWIG (SWIG only for Android Java). `SMILEapi.py` (400+ lines) wraps everything with `CFUNCTYPE` callbacks.

3. **Components have per-module BUILD_FLAGS** already compiled conditionally:
   - `BUILD_COMPONENT_PitchJitter`, `BUILD_COMPONENT_FormantLpc`, etc.
   - Can strip: portaudio/, ffmpeg/, video/, classifiers/, rnn/, functionals/

4. **DSP pipeline**: `cFramer ‚Üí cWindower ‚Üí cTransformFFT ‚Üí cFFTmagphase ‚Üí [pitchACF|formantLpc|melspec|energy|spectral|pitchJitter]` ‚Äî well-factored component hierarchy.

### Verified: ARM64 OpenSMILE API Works

Tested `opensmile.Smile.process_signal()` on Oracle ARM64 with 440Hz test tone:
- 96 frames from 1 second of audio at 16kHz
- 25 eGeMAPS features extracted per frame
- F0 detected: 47.98 semitones (rel. 27.5Hz) = 439.6Hz ‚Üí **A4, spot-on**
- frame_size=1024, hop_size=512 ‚Üí ~46 frames/second at 16kHz

### What This Means For Our Architecture

The optimal bridge is SIMPLER than planned:

    ‚Üí features_to_midi_cc() MIDI CC mapping
    ‚Üí Forward to Ghost Track (:8767) + tminus-dispatcher (:8768)

**Already implemented in `server.py`** ‚Äî the server uses `opensmile.Smile.process_signal()` directly, not subprocess calls.

### Lead-Sheet-MIDI Format
4-track MIDI-compatible format:
1. Transcript (lyrics meta-events, time-synced)
2. Pitch Contour (MIDI notes, prosodic contour)
3. Prosody CC (CC#74=pitch, CC#71=volume, CC#11=energy)
4. Stage Directions (SysEx: speaker_id, role, agreement)

### MIDI as CAM
MIDI CC channels control conversation parameters:
- CC#74 (cutoff) = ternary pitch (+1=bright/rising, -1=dark/falling)
- CC#71 (resonance) = ternary volume
- CC#10 (pan) = speaker position (left=Bob, right=Alice)
- SysEx = speaker ID + dialogic role
