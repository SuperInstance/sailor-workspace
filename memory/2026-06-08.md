## Claude Code Deep Analysis — Game Changer (2026-06-08 19:02 UTC)

**Claude Code subagent completed** (11m46s, 569k tokens) analyzing `/tmp/opensmile` C++ codebase structure. Report: `memory/opensmile-codebase-analysis.md` (24KB).

### Critical Finding: OpenSMILE Already Has What We Need

The pip-installed `opensmile 2.6.0` ships a pre-built `libSMILEapi.so` for ARM64 Linux at:
```
opensmile/core/bin/manylinux_2_17_aarch64/libSMILEapi.so
```

**No custom build or source refactoring is needed.** The entire refactoring plan in `memory/opensmile-refactoring-plan.md` was over-engineered.

### Key Architecture Facts (from codebase analysis)

1. **libSMILEapi.so exposes ~20 C functions** for the full lifecycle:
   - `smile_new()` / `smile_free()` — lifecycle
   - `smile_initialize(configFile, options)` — loads config .ini
   - `smile_extsource_write_data()` — push PCM in
   - `smile_extsink_set_data_callback()` — receive features via callback

2. **Python wrapper uses ctypes directly** — no Cython, no pybind11, no SWIG (SWIG only for Android Java). `SMILEapi.py` (400+ lines) wraps everything with `CFUNCTYPE` callbacks.

3. **Components have per-module BUILD_FLAGS** already compiled conditionally:
   - `BUILD_COMPONENT_PitchJitter`, `BUILD_COMPONENT_FormantLpc`, etc.
   - Can strip: portaudio/, ffmpeg/, video/, classifiers/, rnn/, functionals/

4. **DSP pipeline**: `cFramer → cWindower → cTransformFFT → cFFTmagphase → [pitchACF|formantLpc|melspec|energy|spectral|pitchJitter]` — well-factored component hierarchy.

### Verified: ARM64 OpenSMILE API Works

Tested `opensmile.Smile.process_signal()` on Oracle ARM64 with 440Hz test tone:
- 96 frames from 1 second of audio at 16kHz
- 25 eGeMAPS features extracted per frame
- F0 detected: 47.98 semitones (rel. 27.5Hz) = 439.6Hz → **A4, spot-on**
- frame_size=1024, hop_size=512 → ~46 frames/second at 16kHz

### What This Means For Our Architecture

The optimal bridge is SIMPLER than planned:
```
Browser mic → WS (:8765) → opensmile.Smile.process_signal(audio, sr) 
    → 25 eGeMAPS features (pandas DataFrame) 
    → _parse_features() clean dict 
    → features_to_midi_cc() MIDI CC mapping
    → Forward to Ghost Track (:8767) + tminus-dispatcher (:8768)
```

**Already implemented in `server.py`** — the server uses `opensmile.Smile.process_signal()` directly, not subprocess calls.
